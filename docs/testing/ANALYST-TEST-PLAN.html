Power BI Viewer — Analyst Test Charter v2.1.10 · Prepared for Matt

Part 0. Read this first, Matt (yes, the whole page)

Hi Matt. This is Power BI Viewer — a small desktop app I built so our clients (and you) can open Power BI reports in a clean window without wandering off into the rest of the Microsoft universe and getting lost, which I know is a journey you've made many times.

You have never seen this app before. That is exactly why you're perfect for this: every confusing thing you trip over is a real bug or a real design problem, and you trip over things with a consistency that borders on professional. Version 2.1.10 contains several features that no human being has ever tested — they've only ever been poked at by automated scripts, which, unlike you, follow instructions. Your job is to be the first human.

How to use this charter:

  • Work through the tests in order. The first seven are the never-human-tested tier — they matter most.
  • Each test tells you the purpose, the numbered steps, what PASS looks like, and what FAIL looks like (plus what to write down).
  • Tick the checkbox at the end of each test. Physically. With a pen. The dopamine is free.
  • When something looks wrong: screenshot it (Win+Shift+S), write one line, keep moving. Part 3 explains how to report.
Before you start: install Power BI Viewer, launch it, and sign in with your normal work Microsoft account when the sign-in window appears. After sign-in you'll see a sidebar on the left with five entries: Home, Workspaces, Apps, Insights, and Settings. That sidebar is the whole map. Five buttons, Matt. The Excel ribbon you claim to have mastered has roughly four hundred.
a quick word about the design philosophy. Every button in this app is large, clearly labelled, and does exactly one thing. I want you to know that this wasn't a Microsoft guideline. It was a risk assessment, and the risk was you.

Part 1. The never-human-tested tier (do these first)

Test 1. The Insights board — real data, real clients

Never human-tested~20 min

Purpose: Insights is a data-health board: every client workspace, every dataset and dataflow, whether the refreshes are working, who has access, and what gets used. It has only ever rendered fake test data. You are the first person to see it on the real tenant, so the big question is simple: does it tell the truth?

  1. Click Insights in the sidebar. The page goes dark — near-black with one orange accent. Only this page does that; it's intentional, not a glitch. (Finally a screen that matches your monitor-at-40%-brightness lifestyle, Matt.)
  2. Wait through the spinner ("Checking every dataset and dataflow you can see…"). Note roughly how long it takes.
  3. Check the five summary tiles across the top: Broken, Overdue, Running, Healthy, Workspaces. Tiles with problems are highlighted; quiet tiles stay dim.
  4. Look at the Client health board. Each client workspace is one collapsible group. Broken workspaces should be sorted first and arrive already expanded; healthy ones start folded. Expand a quiet one yourself.
  5. Inside a group, inspect a few rows. Each row shows: a status chip (● OK green, ⬣ Failed red, ▲ Cancelled, ◌ Never run, ◐ Running, ◇ Live), the item name, and a KIND tag (dataset / dataflow).
  6. Find a broken or overdue item (if the tenant is having a good day, congratulations — note that and move on). It should show a red "down 26h"-style duration, or "down — never succeeded".
  7. Study the 12-dot strip on a few rows: the last 12 refresh runs, oldest on the left, newest on the right. Green dot = OK, red = failed, hollow = no history. Hover a dot — it shows the run's time. If any run failed you'll see a red caption like "3 of last 12 runs failed".
  8. Truth check (the important bit): pick 2–3 items and compare against the Power BI web portal (app.powerbi.com → workspace → refresh history). Do status, last-success time, and recent failures actually match?
  9. Check the Who has access section: expand a workspace, confirm the member list and roles look right.
  10. Check Your usage: "You open most" will be thin until you've used the app a while — fine. "Never opened by you" should list real items you can see but haven't opened. For you that list should be magnificent.
  11. Click Refresh (top right). The button reads "Refreshing…", then the snapshot timestamp under the title updates.
PASS looks like Tile counts agree with the board; broken groups first and pre-expanded; dot strips read oldest→newest and match the portal's refresh history; the 2–3 truth-checked items agree with reality; Refresh updates the snapshot time.
FAIL looks like / write down Anything shown green here but failed in the portal (or vice versa) — write the item name, workspace, and both statuses. Also report: dot strip in the wrong order, tile counts that don't add up, a spinner that never ends, or the amber banner "Some workspaces could not be fully read…" (note which workspaces it lists).
this board condenses every client into traffic-light dots so that status can be absorbed in one glance, with zero reading. I designed it long before I knew you'd be the tester, and yet somehow it's the most you-proof thing I've ever made. Fate.

Test 2. Admin unlock — end to end

Never human-tested~10 min

Purpose: at the bottom of Insights there's a locked Admin view — tenant-wide usage ("who opened what, across ALL users") and App audiences. Unlocking it the first time triggers a one-time Microsoft permission window. Nobody has ever clicked this button on a real account. You're the chosen one. Try not to read too much into that.

  1. Scroll to the bottom of Insights → the card titled Admin view — everyone's usage and App audiences.
  2. Read the explanatory text in the card. Plain enough for a first-timer? You're our calibration instrument for "plain", Matt — if you understand it, everyone will.
  3. Click Unlock admin view.
  4. Watch the status text next to the spinner. It changes in honest stages: first "Opening Microsoft consent…", after ~10 s "Reading App audiences…", after ~30 s "Crunching activity log — large tenants can take a couple minutes…". Confirm the text actually progresses.
  5. Known quirk to verify: the Microsoft consent window may open BEHIND the app — much like you at parties. If nothing appears, check the taskbar for a flashing window and bring it forward. Write down whether it appeared in front or hid behind.
  6. While it's loading, test Cancel: click it once. The spinner must stop immediately and the Unlock admin view button must come back. Then click Unlock again.
  7. Approve the Microsoft window when it appears (ticking "consent on behalf of your organization" is fine). It should never ask again afterwards.
  8. When it finishes, verify the three blocks: What's being used (reports, views, people, last viewed), Who's using it (users + views), and App audiences (expand one app, see its member list). Plus a caption line like "Last 2 days · snapshot …" and a working Refresh.
PASS looks like Staged text progresses; Cancel instantly returns the Unlock button; consent appears once and never again; the three data blocks fill with plausible real activity. Also a pass: if your account is not a Fabric admin, the exact friendly message "Power BI says this account is not a Fabric administrator, so the tenant-wide view is unavailable."
FAIL looks like / write down A spinner with no consent window anywhere (even in the taskbar — wait 60 s first); Cancel that doesn't cancel; consent asked twice; empty tables right after real people used reports; or any raw error code instead of a sentence. Write down the stage text at the moment it went wrong, and how long you'd been waiting.
the loading text was deliberately written to entertain the impatient during a two-minute wait. We A/B tested the required patience level against a goldfish and then, with you in mind, rounded down. There's also a Cancel button — the dream is alive that you'll finish something anyway.

Test 3. Export PDF

Never human-tested~10 min

Purpose: every viewer (report, dashboard, app) has an Export PDF button in the toolbar. Reports use Power BI's proper export when the license allows it, and silently fall back to a clean screenshot-PDF of the current view when it doesn't. You should not be able to tell which path ran — only that a good PDF lands on disk.

  1. Open any report (Home or Workspaces → click a report).
  2. Click Export PDF in the top-right toolbar. It's clearly labelled, with a little download icon. This button was designed with you specifically in mind, Matt — it's very large, and it does the entire job with one click.
  3. A save dialog opens. Pick a folder and save. Anywhere but your desktop, Matt — we've all seen your desktop; archaeologists will too, someday.
  4. The button reads "Exporting…" and is disabled. When done, the toolbar briefly shows "Exported to PDF" (~4 s).
  5. Open the saved file. Is it the report you were looking at, readable, not blank, not cropped into abstract art?
  6. Cancel path: click Export PDF again, but press Cancel in the save dialog → toolbar must show "Export cancelled" and nothing is written.
  7. Double-click guard: hammer the Export button twice fast — only ONE save dialog should appear.
  8. Repeat steps 2–5 once on a dashboard and once inside an app (those always export a snapshot of the current view — that's by design).
PASS looks like PDF saved where you chose, opens, matches the screen; "Exported to PDF" / "Export cancelled" messages appear; no double-dialog on a double-click.
FAIL looks like / write down "Export failed" (write the exact message and which report/dashboard), a blank or truncated PDF (keep the file and send it), filter panes visible in the export, or a frozen "Exporting…" that never resolves.

Test 4. Slideshow & kiosk mode — including the escape room

Never human-tested~15 min

Purpose: reports can run as a full-screen slideshow (each report page = a slide) for wall displays. In kiosk mode a single Esc is deliberately ignored so a stray elbow can't kill the lobby screen — exit requires a deliberate gesture. We need to know a first-time human can get in, and crucially, back out.

  1. Open a report with several pages and click Slideshow in the toolbar. It goes full screen.
  2. Test the controls at the bottom: Play/Pause, the ◀ ▶ arrows, and the dot indicators. Keyboard: →/←, Space (next), P (play/pause).
  3. Press Play and don't touch anything: slides should auto-advance, the controls fade out after ~3 s, and the mouse cursor disappears. Wiggle the mouse — everything comes back. A feature, not a haunting.
  4. Click the gear icon (top right) → drag the Interval slider, confirm the pace changes.
  5. Press Esc once → you should drop straight back to the normal report view. (This is manual mode; one Esc works here.)
  6. Now kiosk mode: go to Settings → Slideshow / Presentation Mode, switch Auto-start slideshow ON. Reopen the report and start Slideshow — it should start playing by itself.
  7. Tap Esc once. Nothing should happen. That's correct behaviour, not a bug — resist your instincts, Matt, the urge to file this one is going to be enormous.
  8. Find the small persistent hint in the bottom-right corner: "Hold Esc 3s to exit / or Ctrl+Shift+Q". It must stay visible even after the main controls fade.
  9. Exit gesture 1: hold Esc down for 3 full seconds. Three, Matt. One-Mississippi, two-Mississippi, three — roughly the upper bound of your documented attention span, so dig deep. The slideshow should exit.
  10. Exit gesture 2: start the slideshow again, then press Ctrl+Shift+Q. Instant exit.
  11. Turn Auto-start slideshow back OFF in Settings. (Write that on your hand if needed.)
PASS looks like Auto-advance works at the set interval; controls and cursor fade and return; manual mode exits on one Esc; kiosk mode ignores single Esc but exits on both the 3-second hold and Ctrl+Shift+Q; the corner hint never disappears.
FAIL looks like / write down Trapped in kiosk mode (neither gesture works — note exactly what you pressed and for how long, then Alt+F4 and breathe); single Esc killing kiosk mode; controls that never fade or never come back; slideshow stuck on one slide (note slide name and how long).
if you do get stuck in full screen, do not unplug the computer. I know it's your signature move. There are two documented exits and they are written on the screen at all times, in the corner, in your favourite font: large.

Test 5. Workspaces — the Retry button under real failure

Never human-tested~10 min

Purpose: when a workspace's contents fail to load, the app must say so and offer Retry — instead of lying that the workspace is empty. This failure path has never been triggered by a human. We'll simulate the outage with the one testing tool you've been unknowingly mastering for years: a dropped connection.

  1. Click Workspaces in the sidebar. The list of client workspaces loads. Don't expand anything yet.
  2. Disconnect the Wi-Fi (or pull the network cable). Yes, on purpose this time. I know this is usually something that just happens to you, Matt — savour the control.
  3. Click a workspace you have NOT yet expanded. It should briefly show "Loading content…" and then an amber inline warning: "Could not load this workspace. Check your connection or your permissions." with a Retry button.
  4. Key check: it must NOT show "No reports or dashboards in this workspace." — that message is reserved for genuinely empty workspaces. A failure dressed up as emptiness is the exact bug we're hunting.
  5. Reconnect the Wi-Fi. Wait for it to actually come back (the little icon, Matt — the one you never check).
  6. Click Retry. The contents should now load in place — reports and dashboards with an item-count badge — with no app restart needed.
  7. Bonus: collapse and re-expand that same workspace — it should NOT refetch (no spinner), because it remembers.
PASS looks like Offline expand → clear amber warning + Retry (never the "empty workspace" message); after reconnecting, one click of Retry fully recovers in place.
FAIL looks like / write down The workspace pretends to be empty while offline (screenshot it — top priority bug); Retry does nothing or errors even with network back; the whole app needs a restart to recover. Note workspace name and timing.

Test 6. Search

Never human-tested~5 min

Purpose: one search box covers reports, dashboards, apps, and workspaces. It opens with Ctrl+K. That's two keys, Matt, pressed at the same time. I believe in you — statistically, eventually.

  1. From anywhere in the app, press Ctrl+K. The search dialog should pop open with the cursor already in the box. Also verify the mouse route: click the "Search reports and dashboards…" bar in the title bar at the top.

  3. Use ↑/↓ to move the highlight and Enter to open the selected item. It should land directly in that report.
  4. Reopen search, type a workspace name, click it: you should land on the Workspaces page with that workspace automatically expanded.
  5. Search gibberish ("qzxv"): expect a polite "No results found for 'qzxv'", not a spinner of despair.
  6. Test the little ✕ button to clear the text, and Esc to close the dialog.
PASS looks like Ctrl+K opens with focus in the box; arrow keys + Enter navigate; results open the right item; workspace results auto-expand on the Workspaces page; Esc closes.
FAIL looks like / write down Ctrl+K does nothing (note what screen you were on); an item you can definitely open elsewhere never appears in results (note its name and type); clicking a result opens the wrong thing; an amber partial-failure warning above the results (note its wording).

Test 7. Freshness strip — "✓ Updated" on a real refresh

Never human-tested~15 min + a real dataset refresh

Purpose: the right side of every viewer toolbar shows when the data was last refreshed ("Data refreshed: 06/10/2026 14:32 CST", plus a "Dataflow:" line). When newer data is published while you're looking, an orange "● Newer data available" note appears; after a refresh, a green "✓ Updated" confirmation flashes for ~5 seconds. None of this has ever been watched by human eyes against a real refresh.

  1. Open a report. Find the freshness strip at the right of the toolbar. Immediately after opening it may briefly show "Data refreshed: —" placeholders — they should fill in within seconds.
  2. Sanity-check the timestamp against the dataset's last refresh in the Power BI portal. Note: data older than 24 h shows with an amber ⚠ — correct behaviour, not a bug.
  3. Now trigger a real dataset refresh in the Power BI portal (workspace → dataset → Refresh now) for the dataset behind your open report. Leave the report open in the Viewer.
  4. Wait and watch the strip (a minute or so after the refresh completes). The orange "● Newer data available" note should appear next to the timestamps. This is a waiting game, Matt — the app will blink before you do, but it's going to be close.
  5. Click Refresh in the toolbar. When it completes: a green "✓ Updated" flashes for about 5 seconds, and the timestamps advance to the new refresh time.
  6. Open an app (Apps page) and confirm its strip also shows a live "(N min ago)" age that ticks upward. If the app spans several datasets the label honestly reads "Oldest data" — same for dashboards.
PASS looks like Timestamps match the portal; "● Newer data available" appears within ~a minute of the real refresh finishing; clicking Refresh produces the green "✓ Updated" flash and advanced timestamps; the strip never vanishes outright.
FAIL looks like / write down Timestamp disagrees with the portal (write both); the "newer data" note never shows after a confirmed refresh (note how long you waited); "✓ Updated" never flashes after a successful refresh; placeholders ("—") that never resolve.
this whole feature exists so nobody ever again presents last Tuesday's numbers while calling them "live" — a hypothetical scenario I definitely did not personally witness you star in at the Q3 review.

Test 9. Auto-update — the silent upgrade

Verify~5 min, spread out

Purpose: the app updates itself: it downloads new versions in the background and installs them silently the next time it restarts. No pop-ups, no nagging. It improves quietly, without being asked. Try to learn from it, Matt.

  1. Open Settings → scroll to About. Write down the version shown ("Power BI Viewer v…").
  2. Use the app normally for 15+ minutes (the other tests count — multitasking, but the kind you can handle: zero extra effort).
  3. Close the app fully, reopen it, and check About again. If an update had been published, the version should now read 2.1.10 or higher — and you should NOT have seen any installer windows or progress bars.
  4. If it still shows the old version, restart once more later in the day and re-check (the download may still have been in flight).
PASS looks like Version reaches 2.1.10+ within a restart or two, with zero visible installer fuss. (A dialog saying "A required update is ready" followed by an auto-restart is also a PASS — that's the rare mandatory-update path.)
FAIL looks like / write down Version stuck below 2.1.10 after several restarts across a day (write the version and each restart time), an installer wizard appearing, or the app failing to launch after an update.

Test 14. Error-path spot checks — break it on purpose

Verify~10 min

Purpose: a quick tour of failure states. The rule everywhere: a human sentence and a retry — never a white screen, never raw error gibberish. You break software without trying, Matt; today we're just removing the "without trying".

  1. Wi-Fi off → open a report you haven't opened this session. Expect a readable error and a way to retry — not a blank void.
  2. Wi-Fi off → open Insights fresh. Expect an error message with a Try again button; reconnect and confirm Try again recovers.
  3. Wi-Fi off → press Ctrl+K and search. Note whether it shows the amber partial-results warning or a clean failure (either is acceptable — gibberish is not).
  4. Wi-Fi back on. Settings → Sign out, then sign back in. You should land back on Home, signed in, with content loading normally.
  5. While any report is open, leave the machine alone past your screen-lock time, unlock, and confirm the report still responds (refresh it once).
PASS looks like Every failure shows a plain-English message and a retry path; sign-out/sign-in round-trip is clean; nothing requires killing the app.
FAIL looks like / write down Any white/blank screen (screenshot + what you did last), raw error codes shown to the user, retry buttons that don't retry, or a sign-in loop.

Part 3. Reporting your findings (the Matt-proof method)

For every issue, I need exactly two artifacts:

  • A screenshot. Win+Shift+S, drag a box, done. Of the app, Matt — not a photo of the monitor taken with your phone. We've discussed this. There was a slideshow about it. You were in it.
  • One line of text, in this shape: where I was → what I did → what I saw → what I expected. Example: "Insights → clicked Unlock admin view → spinner for 5 min, no consent window anywhere → expected consent or an error." One line. I know brevity hunts you like a personal enemy, but the format will protect you.

Send everything to Brendan (email or chat — you know where I live, digitally and otherwise). Batch them at the end of each test if you like; do not save them all for a single dramatic reveal on Friday. Also send the PASS list — knowing what works is half the point, and it'll be nice for you to deliver good news for a change.

Severity cheat-sheet: "app lies about data" (wrong freshness, fake-green Insights, empty-looking failed workspace) = drop everything and tell me immediately. "Button mislabelled or ugly" = one line, end of day. "I personally find orange suspicious" = that's not a bug, Matt, that's the brand.
